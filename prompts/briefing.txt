TASK — CHART CLASSIFICATION + PILOT BRIEFING

FIRST, output a JSON line classifying which charts show relevant weather
for this specific flight route ({origin} → {destination}).
A chart is "relevant" only if the weather meaningfully impacts the planned route, \
departure/arrival airports, or practical alternates at the planned time window.

Charts showing no meaningful weather impact along the route go in the reference pile.

Output this EXACT format on the FIRST line (no markdown code fences):
SIGNIFICANT_CHARTS: ["label1", "label2", ...]

All chart labels:
{label_list}

THEN a blank line, then the pilot briefing below.

---

REQUIRED SECTIONS (use these exact h2 headings):

<h2>Executive Summary</h2>
One short paragraph with exactly:
a) What kind of day this is (e.g., "This is a VMC wind/terrain day, not a weather system problem.")
b) Primary Risk: one sentence
c) GO / MARGINAL GO / NO-GO
d) The single most important trigger to monitor

Do not expand beyond these four elements. A pilot should read only this and know the call.

<h2>The Day Before — What to Watch</h2>
Short practical re-check list for the evening before.
What to look for and what would change the go/no-go.

<h2>Leaving {origin} — Departure at {dep_str}</h2>
Describe departure ceiling/visibility/wind. State whether this is:
- VMC departure
- brief layer penetration
- likely departure IMC (only if TAFs show BKN/OVC at or below climb altitudes)

Use pilot language, not METAR codes.

<h2>The Ride at {altitude} ft</h2>

WINDS ALOFT:
If winds/temps data is provided, decode and present actual winds at cruise altitude
for stations along the route. Calculate headwind/tailwind/crosswind components.

RIDE QUALITY:
Classify using standard terms only:
smooth / light chop / light turbulence / moderate turbulence / severe turbulence

Wind-over-terrain reference:
- <20 kt: minimal terrain effects
- 20-30 kt: light to moderate terrain turbulence possible
- 30-40 kt: mountain wave likely, moderate turbulence possible
- >40 kt: moderate or greater turbulence likely

ICING:
Include a vertical icing table: Altitude | Icing Prob | Notes
State: freezing level, worst icing band, SLD risk (or explicitly no SLD signal).

ALTITUDE STRATEGY:
State explicitly:
a) best altitude band and why
b) worst altitude band and why
c) escape option if conditions worsen: climb, descend, divert, or turn around

HAZARD TABLE:
Hazard | Risk | Leg | What to Expect
Incorporate icing (FIP), turbulence (GTG), G-AIRMET, SIGMET, SigWx, and QPF findings.

<h2>Getting Into {destination}</h2>
Describe expected arrival conditions. Estimate block time.
Include: runway wind estimate with uncertainty, practical alternates.
Distinguish: non-event VMC arrival / terrain-localized issue / broader weather issue.

For terrain airports: do not assume winds aloft = runway winds.
State uncertainty explicitly.

<h2>Fly or No</h2>
Verdict first: GO / MARGINAL GO / NO-GO.
Hard stops, then watch items.

If recommending IFR, state why: IMC, terrain complexity, or workload/safety margin.

<h3>What changes this call</h3>
- 2-4 bullets max
- at least one quantitative trigger (winds > X kt, ceilings < X ft)

<h2>If You Go — Do This</h2>
Concrete action items only: altitude, alternates, fuel, what to check morning-of.

If departure timing materially changes risk (convection, frontal passage, diurnal winds,
marine layer), state best and worst windows with a one-sentence reason each.
If timing is not a factor, omit.

<h2>Confidence</h2>
State High / Medium / Low and what forecast elements could still change the call.
Do not assign High if terrain-driven effects dominate or critical data is missing.

FACTUAL ACCURACY:
Do not guess airport characteristics (towered/non-towered, airspace class, runway
count, elevation) unless you are certain. If unsure, omit rather than state incorrectly.
Use only the airport names and data provided in the FLIGHT header.

Respond with HTML using: h2, h3, h4, p, ul, ol, li, strong, em, blockquote, table, thead, tbody, tr, th, td, code, hr.
No html/head/body/style/script tags.
